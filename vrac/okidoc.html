Instance personnelle – DIY & soutien non-financier

Créer ta propre instance plutôt que juste donner des sous

Tu n’es pas obligé·e de financer pour soutenir. Tu peux cloner le kit, le faire vivre chez toi, dans ton asso, dans ton quartier, et renvoyer du feedback, des améliorations, des histoires.

Cette page est un guide clé en main pour fabriquer ta propre instance : une clé USB, un dossier sur ton ordi, un petit site chez ton hébergeur ou sur GitHub Pages. L’idée : multiplier les micro-agences de renseignement citoyen, pas centraliser tout chez une seule personne.

Principe de base : tu as le droit moral de copier, adapter, détourner ce kit tant que tu respectes les conditions de non-appropriation crade (pas de revente purement opportuniste, pas de bullshit autoritaire). En cas de doute : explique ce que tu veux faire, c’est déjà une forme de soutien.

1. Prérequis & philosophie de ton instance

Avant de parler fichiers, deux idées :

  • Tu n’es pas un “client” mais un·e co-auteur·ice potentiel·le.
  • Ta version locale a le même droit d’exister que la version “officielle”.

Techniquement, tu as besoin de très peu de choses :

  • Un navigateur récent (Firefox, Chrome, Brave, etc.).
  • Une clé USB ou un simple dossier sur ton ordinateur.
  • Un éditeur de texte simple (Notepad++, VS Code, ou même le bloc-notes si tu n’as que ça).
  • (Optionnel) Un compte GitHub ou un petit hébergeur pour rendre ton instance accessible en ligne.
Étape 1 – Récupérer le kit
MODE : copier-coller assumé
▾

1. Récupère le dossier complet du projet (que tu as sur une clé, sur GitHub, ou par une archive .zip). Dedans, tu devrais trouver au minimum :

  • index.html (le portail principal)
  • SAS.html, carte-0.html, test-0.html, etc.
  • Un ou plusieurs dossiers /pdf, /images, /media selon ton organisation

2. Copie ce dossier tel quel sur :

  • une clé USB (ex : /CLE-VEILLE/instance-tes-initiales/), ou
  • un dossier sur ton ordi (ex : Documents/veille-citoyenne/instance-locale/).

3. Double-clique sur index.html pour vérifier que tout s’affiche correctement dans ton navigateur.

Étape 2 – Choisir ton mode d’installation
MODE : hors-ligne ou en ligne
▸

Tu as trois niveaux possibles :

Clé USB / hors-ligne

  • Tu emportes la clé partout : réunion, atelier, formation, café citoyen.
  • Tu lances les tests / pages directement depuis la clé.
  • Idéal pour les lieux sans bonne connexion internet.

Attention : toujours garder une copie de sauvegarde sur un autre support, au cas où la clé meurt.

Dossier local partagé

  • Tu mets le dossier dans un espace partagé (ex : Nextcloud, réseau interne, clé “commune”).
  • Plusieurs personnes peuvent ouvrir les fichiers HTML et jouer avec le kit.
  • Pratique pour une équipe de base (2–10 personnes).

Cocher dans ta tête : “Qui a le droit de modifier les fichiers ? Qui garde une copie sûre ?”

Petit site web (GitHub Pages / hébergeur mutualisé)

  • Tu crées un dépôt GitHub (ou un dossier sur ton hébergement).
  • Tu y envoies tous les fichiers (index.html et compagnie).
  • Tu actives GitHub Pages ou pointes ton nom de domaine / sous-domaine vers ce dossier.

Résultat : ton instance devient accessible à n’importe qui via une simple URL (https://toncompte.github.io/ton-instance par exemple).

Étape 3 – Personnaliser ta version (sans tout casser)
MODE : retouche en douceur
▸

Commence par des modifications simples :

  • Dans about.html : remplace le texte “qui sommes-nous” par ton contexte.
  • Dans index.html : ajoute ton nom de cellule, ton quartier, ton collectif.
  • Évite au début de modifier trop de styles : teste d’abord que tout fonctionne.

Astuce : fais une copie du fichier avant de le modifier (ex : index-original.html) pour toujours pouvoir revenir en arrière.

Étape 4 – Ajouter tes propres dossiers & tests
MODE : co-auteur
▸

Tu peux :

  • Créer test-local-1.html en copiant un test existant et en changeant les questions.
  • Ajouter tes propres PDF dans un dossier /pdf et les lier depuis une page.
  • Créer une nouvelle “phase” ou un nouveau “jalon” pour ton groupe.

Tant que tu restes cohérent·e avec l’esprit global (éducation populaire, esprit critique, humour, non-toxicité), ta version enrichit l’écosystème.

Checklist “Mon instance tourne pour de vrai”

Si tu coches 3–4 cases : tu as déjà une instance fonctionnelle. À partir de là, tu peux soutenir le projet autrement que par l’argent.

2. Soutenir sans payer : modes “contribution”

Créer ta propre instance, c’est déjà un soutien. Mais tu peux aller plus loin sans forcément sortir la carte bancaire.

Modes de soutien non-financiers

Tu peux combiner plusieurs options. Le but n’est pas d’en faire trop, mais de choisir ce qui te ressemble.

  • Retour d’usage : raconter ce qui a marché / bloqué dans tes tests.
  • Relecture & correction : corriger des coquilles, clarifier des passages.
  • Création de tests locaux : adapter les quizz à ta commune, ton secteur, ton public.
  • Animation : organiser une séance avec un groupe, et en faire un mini-compte-rendu.
  • Relais : faire connaître le kit à d’autres collectifs, syndicats, associations.
  • Technique : aider à héberger, documenter, structurer les fichiers.

Même si tout reste hors radar officiel, ton instance fait partie d’un réseau d’expériences.

Niveau de soutien non-financier envisagé 1 / 5
1/5 – Tu es en mode “je teste pour moi, je regarde ce que ça donne”. C’est déjà mieux que 99 % des gens qui ne tenteront jamais.

Comment donner du feedback utile ?

Tu peux décider d’un canal de contact (mail, formulaire, dépôt GitHub, etc.) et envoyer :

  • Une phrase sur ce qui t’a accroché (ou pas).
  • Une capture d’écran d’un bug ou d’un texte incompréhensible.
  • Une proposition de test, de dossier, de mise en scène (même à l’état de brouillon).
  • Une question : “Est-ce que je peux faire X avec mon instance ?”

Chaque retour, même minuscule, aide à ajuster le projet pour les suivantes et suivants.

3. Articuler ta version avec le reste du projet

Ton instance n’est pas “en concurrence” avec l’originale. Elle peut :

  • Tester des variantes de tests (plus doux, plus trash, plus localisés).
  • Explorer des publics différents (ados, militants, usagers de services, etc.).
  • Servir de preuve quand tu expliques à quelqu’un que “c’est faisable chez soi”.

Si tu le souhaites, tu peux mentionner que ton instance est une fork citoyenne d’un kit plus large, et renvoyer vers la source d’origine pour celles et ceux qui veulent remonter la piste.
